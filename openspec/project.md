# Project Context: Domestic RTP Payment Service API

## Project Purpose

This project exposes domestic real-time payment capability as a Banking-as-a-Service and embedded banking API. The API is designed for corporate systems, fintech backends, ERP platforms, treasury applications, and partner systems that need to create a single domestic real-time payment instruction and query its processing status.

The MVP uses JSON payloads, JWT Bearer authentication, scope validation, idempotency, asynchronous status tracking, downstream mock processing, OpenAPI documentation, Postman support, and GKE Gateway API or Kubernetes Ingress exposure.

## Glossary

- **Client**: The external API caller authenticated by the JWT `sub` claim, such as a corporate system, fintech backend, ERP platform, treasury application, or partner system.
- **Tenant**: The partner, corporate customer, or embedded banking tenant represented by `tenant_id` when the API client serves multiple customers or organizations.
- **Actor**: The entity inside the external client system that triggered the payment action. The actor may be a system process, corporate user, embedded application, or service account.
- **Authorization context**: The internal context derived from validated JWT claims, scopes, tenant information, actor information, token metadata, and correlation ID.
- **Payment instruction**: The JSON business request that describes one domestic real-time payment to be created by `POST /v1/domestic-payments`.
- **Payment ID**: The unique identifier generated by the Payment Service API for an accepted payment instruction.
- **Payment status**: The current lifecycle state for a payment, such as `ACCEPTED`, `PROCESSING`, `COMPLETED`, `REJECTED`, `FAILED`, or `TIMEOUT`.
- **Idempotency key**: The client-supplied `Idempotency-Key` header used to prevent duplicate payment creation for repeated requests.
- **Idempotency record**: The stored record that links an authenticated client, idempotency key, request fingerprint, original accepted response, and payment status reference.
- **Request fingerprint**: A stable representation of the normalized payment creation request body and relevant client context used to detect idempotency replay or conflict.
- **Correlation ID**: The tracing identifier accepted from a request header or generated by the API, then propagated through responses, logs, records, and downstream calls.
- **Downstream payment processor mock**: The local mock implementation that simulates payment processor outcomes for success, business rejection, timeout, and internal failure.
- **Gateway-like controls**: Authentication, scope validation, correlation ID handling, request validation, and consistent error handling performed inside the Payment Service API for the MVP.
- **GKE exposure layer**: GKE Gateway API or Kubernetes Ingress configuration used to expose the MVP API without a custom lightweight API Gateway service.

## Architectural Style

Use a layered Payment Service API architecture:

- Controllers and API adapters handle HTTP routing, headers, request parsing, response mapping, and OpenAPI alignment only.
- Application services or use cases orchestrate payment creation, status query, authentication context usage, idempotency decisions, downstream mock calls, and persistence coordination.
- Domain models represent payment concepts, payment identifiers, idempotency records, authorization context, correlation IDs, and the payment status lifecycle.
- Persistence adapters implement repository interfaces for payment status records and idempotency records.
- Downstream payment processor integrations sit behind a client or gateway interface, with the MVP using a local mock implementation.
- Infrastructure contains framework, configuration, database, mock, OpenAPI rendering, Postman, Kubernetes, GKE Gateway API, and Ingress concerns.

Dependencies must point inward. Domain code must not import or depend on HTTP frameworks, JWT libraries, OpenAPI tools, Postman artifacts, Kubernetes resources, GKE Gateway API, Ingress, database drivers, or downstream mock implementation details.

## Module Boundaries

- Controllers must not access databases, caches, idempotency stores, or downstream clients directly.
- Controllers call application services; application services depend on domain types and interfaces.
- Repository interfaces belong with application or domain contracts; repository implementations belong in infrastructure.
- Downstream payment processor interfaces belong with application contracts; mock and future real implementations belong in infrastructure.
- Authentication, scope validation, correlation ID handling, idempotency decisions, and error mapping must be centralized and reusable.
- Idempotency storage and payment status storage must remain separate concepts, even if both are backed by in-memory or lightweight persistence in the MVP.
- Mock-only controls must not become production payment semantics.
- The downstream payment processor mock must be replaceable with a future real payment integration without changing the public API contract.

## Naming Conventions

- Endpoint paths use kebab-case, for example `/v1/domestic-payments`.
- Domain types use payment-specific names such as `PaymentInstruction`, `PaymentStatus`, `PaymentId`, `IdempotencyRecord`, `AuthorizationContext`, and `CorrelationId`.
- Operation names should be explicit, such as `CreateDomesticPayment` and `GetDomesticPaymentStatus`.
- Payment creation authorization scope is `payments:create`.
- Payment status query authorization scope is `payments:read`.
- JWT audience is `domestic-payment-api`.
- Required JWT claims are `sub`, `scope`, `aud`, `iss`, and `exp`.
- Optional but supported Banking-as-a-Service and embedded banking JWT claims are `tenant_id`, `actor`, `iat`, and `jti`.

## Security and Authentication Constraints

All external payment endpoints require JWT Bearer authentication and scope validation. The Payment Service API derives authorization context from validated JWT claims before payment orchestration.

The JWT primarily represents the calling external client, such as a corporate system, fintech backend, ERP platform, treasury application, or partner system. Optional tenant and actor claims support embedded banking audit and future delegated authorization models.

The authorization context should be available to:

- payment orchestration
- idempotency handling
- audit logging
- downstream payment processor mock calls
- future micro-authorization extension points

Do not log full account identifiers, sensitive account data, raw sensitive payment payloads, bearer tokens, signing keys, or secrets. Mask or omit sensitive values at logging boundaries.

## Idempotency and Payment Processing Constraints

All payment creation requests must include `Idempotency-Key`.

The idempotency behavior is client-scoped:

- Same authenticated client, same `Idempotency-Key`, and same request body returns the original accepted response.
- Same authenticated client and same `Idempotency-Key` with a different request body returns `409 Conflict`.
- Different authenticated clients may use the same `Idempotency-Key` without colliding.

Payment creation returns `202 Accepted` after the request is accepted for processing. It does not represent final payment completion or settlement. Final or terminal outcomes are observed through payment status query.

The MVP payment status lifecycle includes `ACCEPTED`, `PROCESSING`, `COMPLETED`, `REJECTED`, `FAILED`, and `TIMEOUT`.

The downstream payment processor mock must support deterministic success, business rejection, timeout, and internal failure scenarios.

## Observability and Error Handling

Every request must accept an inbound correlation ID or generate one when absent.

Correlation ID must flow through:

- API responses
- JSON error responses
- structured logs
- payment status records
- idempotency records
- downstream mock requests

External APIs must return consistent JSON error responses with stable error code, message, HTTP status, and correlation ID. Validation errors should include field-level details when applicable.

Status query behavior must not leak unrelated client payment data.

## Developer Support and Testing Standards

OpenAPI 3.0.3 documentation must stay aligned with behavior specs and implementation behavior. Swagger UI, Redoc, or equivalent tooling should render the local API documentation.

Postman collection and environment files must stay aligned with the OpenAPI contract, local JWT/token variables, idempotency key behavior, correlation ID handling, payment ID capture, and downstream mock scenarios.

Automated tests should cover:

- OpenAPI contract behavior
- JWT authentication and required claims
- `payments:create` and `payments:read` authorization
- BaaS authorization context derivation
- idempotency replay and conflict behavior
- client-scoped idempotency
- payment status lifecycle
- downstream mock success, rejection, timeout, and failure
- consistent JSON errors
- correlation ID propagation
- sensitive account data masking

Run `openspec validate add-domestic-rtp-payment-service-api` after changing OpenSpec artifacts for this feature.

## Out-of-Scope Guardrails

Do not add the following unless a future OpenSpec change explicitly expands scope:

- cross-border payments
- batch payments
- recurring or standing instructions
- payment cancellation or amendment
- webhooks or callbacks
- ISO 20022 XML ingestion
- encrypted ISO 20022 `pacs.008` payloads
- real core banking integration
- real payment scheme connectivity
- real clearing or settlement
- real balance checks
- real fraud screening
- real sanctions screening
- real FX conversion
- production-grade database design
- production-grade key management or secret rotation
- custom lightweight API Gateway service for the GKE-based MVP

The GKE-based MVP is exposed through GKE Gateway API or Kubernetes Ingress. The architecture should preserve a future path toward Apigee, Kong, durable persistence, real downstream payment integration, and production-grade security controls.
