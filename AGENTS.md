# Project Architecture Constraints

These instructions apply to the entire repository. Follow them when generating, modifying, reviewing, or organizing code for the Domestic RTP Payment Service API.

## Project Vocabulary

- **Client**: The external API caller authenticated by the JWT `sub` claim, such as a corporate system, fintech backend, ERP platform, treasury application, or partner system.
- **Tenant**: The partner, corporate customer, or embedded banking tenant represented by `tenant_id` when the API client serves multiple customers or organizations.
- **Actor**: The entity inside the external client system that triggered the payment action, such as a system process, corporate user, embedded application, or service account.
- **Authorization context**: Internal context derived from validated JWT claims, scopes, tenant information, actor information, token metadata, and correlation ID.
- **Payment instruction**: The JSON business request that describes one domestic real-time payment to be created by `POST /v1/domestic-payments`.
- **Payment ID**: The unique identifier generated by the Payment Service API for an accepted payment instruction.
- **Idempotency record**: Stored record linking authenticated client, `Idempotency-Key`, request fingerprint, original accepted response, and payment status reference.
- **Correlation ID**: Tracing identifier accepted from a request header or generated by the API, then propagated through responses, logs, records, and downstream calls.
- **Downstream payment processor mock**: Local mock implementation that simulates payment processor outcomes for success, business rejection, timeout, and internal failure.
- **Gateway-like controls**: Authentication, scope validation, correlation ID handling, request validation, and consistent error handling performed inside the Payment Service API for the MVP.

## Architectural Style

- Use a layered Payment Service API architecture.
- Keep HTTP controllers and API adapters limited to routing, headers, request parsing, response mapping, and OpenAPI alignment.
- Put business orchestration in application services or use cases.
- Put payment concepts, status lifecycle rules, and value objects in the domain layer.
- Put persistence behind repository interfaces.
- Put downstream payment processor behavior behind a client or gateway interface.
- Keep configuration, framework code, database drivers, Kubernetes manifests, OpenAPI rendering, Postman artifacts, and mock infrastructure outside domain logic.
- Depend inward: infrastructure may depend on application and domain abstractions; domain must not depend on infrastructure.

## Module Boundaries

- Do not let controllers access databases, caches, idempotency stores, or downstream clients directly.
- Do not let domain code depend on HTTP, JWT libraries, OpenAPI, Postman, Kubernetes, GKE Gateway API, Ingress, Swagger UI, Redoc, or downstream mock implementation details.
- Centralize authentication, scope validation, correlation ID handling, idempotency decisions, and error mapping.
- Keep idempotency storage separate from payment status storage, even if both use in-memory or lightweight persistence in the MVP.
- Keep the downstream payment processor mock swappable with a future real payment integration.
- Keep mock-only controls out of production payment semantics.

## Naming Conventions

- Use kebab-case for endpoint paths, for example `/v1/domestic-payments`.
- Use payment-specific domain names such as `PaymentInstruction`, `PaymentStatus`, `PaymentId`, `IdempotencyRecord`, `AuthorizationContext`, and `CorrelationId`.
- Use explicit operation names such as `CreateDomesticPayment` and `GetDomesticPaymentStatus`.
- Use `payments:create` for payment creation authorization.
- Use `payments:read` for payment status query authorization.
- Use the JWT audience value `domestic-payment-api`.
- Use JWT claims `sub`, `scope`, `aud`, `iss`, and `exp` as required claims.
- Treat JWT claims `tenant_id`, `actor`, `iat`, and `jti` as optional but supported BaaS and embedded banking context.

## Banking and Security Constraints

- All payment creation must require `Idempotency-Key`.
- Repeated requests with the same authenticated client, same `Idempotency-Key`, and same request body must return the original accepted response.
- Reusing the same authenticated client and `Idempotency-Key` with a different request body must return `409 Conflict`.
- JWT Bearer authentication and scope validation are mandatory for all external payment endpoints.
- The Payment Service API must derive authorization context from validated JWT claims before payment orchestration.
- Do not log full account identifiers, sensitive account data, raw sensitive payment payloads, bearer tokens, signing keys, or secrets.
- Mask or omit sensitive values at logging boundaries.
- Pass authorization context to the downstream payment processor mock for future micro-authorization support.

## API and Error Handling Constraints

- External APIs must use JSON request and response payloads.
- External APIs must return consistent JSON error responses with stable error code, message, HTTP status, and correlation ID.
- Validation errors should include field-level details when applicable.
- Every request must accept an inbound correlation ID or generate one when absent.
- Correlation ID must flow through responses, logs, idempotency records, payment status records, and downstream mock calls.
- Payment creation must return `202 Accepted` after request acceptance; final or terminal payment outcomes are observed through status query.
- Status query must not leak unrelated client payment data.

## MVP Scope Guardrails

- Support domestic real-time payments only.
- Support one payment instruction per API request.
- Support JSON business payloads only.
- Do not add cross-border, batch, recurring, standing instruction, cancellation, amendment, webhook, callback, ISO 20022 XML, encrypted `pacs.008`, real core banking, real payment scheme connectivity, clearing, settlement, balance check, fraud screening, sanctions screening, or FX conversion unless a future OpenSpec change explicitly adds it.
- Expose the GKE-based MVP through GKE Gateway API or Kubernetes Ingress.
- Do not implement a custom lightweight API Gateway service for the MVP.
- Keep the future path open for Apigee, Kong, durable persistence, real downstream payment integration, and production-grade key management.

## Developer Support and Verification

- Keep OpenAPI 3.0.3 documentation aligned with behavior specs.
- Keep Postman collection and environment files aligned with the OpenAPI contract and mock scenarios.
- Provide deterministic downstream mock scenarios for success, rejection, timeout, and internal failure.
- Add focused contract and integration tests for authentication, authorization, idempotency, payment lifecycle, downstream mock outcomes, error handling, and correlation ID propagation.
- Run `openspec validate add-domestic-rtp-payment-service-api` after changing OpenSpec artifacts for this feature.
